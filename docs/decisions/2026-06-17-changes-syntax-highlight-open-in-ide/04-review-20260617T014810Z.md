## Security / correctness
- **XSS guard preserved.** The only HTML injected via `dangerouslySetInnerHTML` comes
  from `highlightDiffLine`, which always escapes (hljs self-escapes; fallback escapes
  the five significant chars). Unit test asserts `<script>` → `&lt;script&gt;` with no
  literal tag.
- **Injection.** GUI launch passes the file path as a positional `"$1"` (never
  interpolated); the editor command comes from the trusted `ide` setting. The vim
  bootstrap shell-quotes the path.
- **PATH trap.** GUI editors launch through `$SHELL -lc` (login shell), inheriting the
  user's real PATH — matches `runHook`/`commandExists`; no `fix-path` dep.
- **Terminal editor.** vim is routed into a real pty (in-app shell session), not a
  bare `spawn` from main (which would have no TTY).

## Risks / follow-ups (non-blocking)
- **Bundle size.** The full highlight.js bundle lands in the renderer chunk (~2.7 MB
  total js). Acceptable for a desktop Electron app; if it matters later, switch to
  `highlight.js/lib/core` + per-language registration.
- **Per-line highlighting** (by design) can mis-colour multi-line constructs (open
  block comments, unterminated template strings) — Out of Scope for v1.
- **DiffPane reads `store.canOpenInIde()` without subscribing** — the icon's
  enabled/disabled state refreshes on the pane's existing load/auto-refresh rather
  than instantly on a settings change. Acceptable; settings are seeded at launch in tests.
- **No line targeting** (`-g file:line`) on GUI launch — Out of Scope for v1.
